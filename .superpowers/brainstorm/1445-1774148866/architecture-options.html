Which architecture approach fits best?

Three ways to structure the Chrome Extension internals

A

All-in-One Side Panel Recommended

Everything lives in the React side panel: file parsing (PDF.js / mammoth / xlsx), Gemini API calls, template storage, and form generation. One stable context, no message passing.

Pros

  • Simplest codebase
  • Stable context (no SW termination)
  • Easy to debug
  • Full DOM access everywhere

Cons

  • All logic in one place
  • No background tasks
B

Side Panel + Service Worker

React side panel handles UI and file parsing. Service worker handles Gemini API calls and storage logic. Message passing between them.

Pros

  • Standard MV3 pattern
  • API key never exposed in panel
  • Clean separation of concerns

Cons

  • SW can terminate mid-request (MV3 bug)
  • Message passing boilerplate
  • Harder to debug
C

Side Panel + Offscreen + Service Worker

Three separate contexts: UI in panel, heavy file processing in an offscreen document, Gemini calls via service worker. Maximum isolation.

Pros

  • Maximum isolation
  • Theoretically cleanest

Cons

  • 3-way message passing complexity
  • SW termination still a risk
  • Overkill for single-user tool
  • Much harder to build and maintain

My recommendation: Option A — for a single-user local tool with user-triggered actions only, the added complexity of B or C solves problems you won't have. The side panel is a stable, persistent context that handles everything cleanly.
